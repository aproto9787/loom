Short answer

지금 제일 좋은 선택

  • 1순위: 최종 검증 재실행
  • 2순위: 하나의 feature commit 생성
  • 3순위: 원격 브랜치 push
  • 4순위: draft PR로 통합 검토
  • 5순위: merge 후 main checkout에서 pull + Heddle 재시작

현재 바로 main에 merge하지 않는 이유는 변경량이 크고 default workflow가 바뀌기 때문입니다.

Mental model

실제 운용 감각

1. 요청 해석low-risk인지 code-change인지 side-effect인지 먼저 나눕니다.
2. tier 선택quick/code/side_effect/enterprise 중 하나로 manifest에 기록합니다.
3. worker 실행필요한 경우 Codex worker/reviewer/user-advocate를 Heddle MCP로 위임합니다.
4. gate 기록heddle_record_gate로 pass/fail/blocked 상태를 남깁니다.
5. Studio 확인run detail에서 tier, gate, approval, rollback 상태를 봅니다.
Rollout recommendation

워크플로우를 바꿀 만한가

  • 예. 현재 구현은 Heddle의 기본 workflow로 바꿀 만합니다.
  • 단, main checkout에 수동으로 복사하지 말고 PR merge 후 pull로 반영하십시오.
  • 변경 후에는 실행 중인 Heddle MCP/server/studio 세션을 재시작해야 새 default flow와 tool surface가 확실히 반영됩니다.
